01 — System Overview & Architecture
RS³ Films is a three-layer platform: an Android app consumed by fans, a React web admin dashboard operated by the team, and a Node.js REST API that sits between them. All business logic runs on the server. The app and dashboard are purely rendering clients.

Three Products. One API.
Android App (Kotlin + Jetpack Compose)  — fan-facing mobile experience
Admin Web Dashboard (React)             — full content and rule management
REST API (Node.js + Express)            — single backend serving both clients
All three share the same database and media infrastructure.

How the Three Systems Talk
Client	Communicates With	How
Android App	REST API	HTTPS + JWT Bearer token in Authorization header. Refresh token stored in Android Keystore.
Admin Dashboard	REST API	HTTPS + JWT in HttpOnly cookie (admin role claim). Separate /admin/* route namespace.
REST API	PostgreSQL	Prisma ORM — parameterized queries, schema migrations, connection pooling via PgBouncer.
REST API	Cloudinary	SDK calls for image/video upload, signed URL generation, transformations.
REST API	Firebase	Admin SDK: OTP verification, push notifications (FCM), dynamic link creation.
REST API	Redis	Rate limiting, short-TTL cache (app_content, offers), BullMQ job queue.
 
02 — Media & Storage Strategy
PostgreSQL is the right choice for structured relational data: users, referral chains, coin ledgers, rules, and metadata. It is the wrong choice for storing or serving binary media. Videos especially must never be stored in or served through PostgreSQL.

Why PostgreSQL Is Wrong for Video & Images
Do NOT store images or videos in PostgreSQL
Binary columns (BYTEA) bloat the DB, slow backups, and make replication expensive.
PostgreSQL has no CDN, no adaptive bitrate streaming, no thumbnail generation.
Every media request going through the DB server consumes DB connections — a scarce resource.
Serving a 50 MB video from PostgreSQL to 100 concurrent users = DB under severe IO pressure.
Recovery time after incidents is much longer when the DB contains gigabytes of binary data.

Recommended Storage Stack — Two-Database Architecture
Use PostgreSQL for everything relational (metadata, IDs, URLs). Use a dedicated media platform for all files.

Store	What it holds	Service	Why
Relational DB	Users, rules, transactions, metadata, URLs	PostgreSQL 16	ACID, joins, foreign keys, Prisma ORM support
Image CDN	Selfies, feed images, offer banners, splash image	Cloudinary	Auto-resize, WebP conversion, signed URLs, CDN delivery, free tier generous
Video CDN	Feed reels, behind-the-scenes clips	Cloudinary Video	Adaptive bitrate HLS, thumbnail extraction, MP4/WebM, same API as images
Document Cache	app_content key-values, offer list, feed metadata	Redis 7	Sub-millisecond read, avoids repeated DB queries for unchanged content
Full-text search	Feed post search (future)	Algolia / pg FTS	Optional. Start with PostgreSQL FTS, migrate to Algolia at scale.

Cloudinary — Why It Is the Right Choice
•	Single SDK for both images and videos. Same upload API, same transformation pipeline.
•	Adaptive Bitrate Streaming: Cloudinary auto-generates HLS (.m3u8) from uploaded MP4. ExoPlayer in the app plays it natively — smooth streaming even on slow connections.
•	Automatic image optimisation: upload a 4K photo; Cloudinary delivers it resized + WebP to Android.
•	Signed URLs: media never exposed publicly. Every URL has a signature + expiry. Admin-controlled access.
•	Transformation on-the-fly: splash image delivered at exact screen resolution. Feed thumbnails generated automatically.
•	Free tier: 25 GB storage + 25 GB bandwidth/month. More than sufficient for launch.
•	Backup: Cloudinary backs up originals. You never lose a file.

Media Flow — How it Works End-to-End
01	Admin uploads image or video from the Admin Dashboard (drag & drop).
02	Browser sends the file directly to Cloudinary via a signed upload preset (no API server involved — reduces latency and server load).
03	Cloudinary returns the permanent public_id and secure_url.
04	Dashboard sends the secure_url to the API (POST /admin/feed or PATCH /admin/app-content/:key).
05	API stores only the URL string in PostgreSQL. Zero binary data in the DB.
06	App calls GET /feed. API returns the URL from PostgreSQL.
07	App loads the image/video directly from Cloudinary CDN — API server is not in the media path.
08	For video: Cloudinary URL is an HLS manifest. ExoPlayer streams it adaptively.

Cloudinary Setup (one-time)
Create account at cloudinary.com (free tier).
Create two upload presets: rs3-admin (signed, for admin dashboard uploads) and rs3-user (signed, for selfie uploads).
Set CLOUDINARY_CLOUD_NAME, CLOUDINARY_API_KEY, CLOUDINARY_API_SECRET in server .env.
Install SDK: npm install cloudinary.
For video: enable Adaptive Streaming add-on (free tier available).

Critical Service: Referral Chain Resolution
This is the most important server-side function. It runs exactly once per new registration. The chain depth is dynamic — read from point_rules, not hardcoded.

// referralService.ts — core chain resolution
async function resolveReferralChain(newUserId, referredByUserId, prisma) {
  // 1. Get active referral levels from DB
  const levels = await prisma.pointRule.findMany({
    where: { rule_type: 'referral_level', active: true },
    orderBy: { level_number: 'asc' }
  });
  // 2. Walk up the tree — one step per active level
  let currentUserId = referredByUserId;
  for (const level of levels) {
    if (!currentUserId) break;
    // Insert chain row
    await prisma.referralChain.create({
      data: { referrer_id: currentUserId, referee_id: newUserId,
              level: level.level_number, reward_coins: level.coins,
              status: 'pending' }
    });
    // Move to next level up
    const parent = await prisma.user.findUnique({
      where: { id: currentUserId }, select: { referred_by_user_id: true }
    });
    currentUserId = parent?.referred_by_user_id ?? null;
  }
}

// Called after OTP verification — releases pending coins
async function releaseReferralCoins(newUserId, prisma) {
  const chains = await prisma.referralChain.findMany({
    where: { referee_id: newUserId, status: 'pending' }
  });
  for (const chain of chains) {
    await prisma.$transaction([
      prisma.coinTransaction.create({ data: {
        user_id: chain.referrer_id, type: 'earned_referral_l'+chain.level,
        coins: chain.reward_coins, reference_id: chain.id,
        note: 'Referral reward — Level '+chain.level
      }}),
      prisma.user.update({ where: { id: chain.referrer_id },
        data: { coin_balance: { increment: chain.reward_coins } }
      }),
      prisma.referralChain.update({ where: { id: chain.id },
        data: { status: 'awarded', rewarded_at: new Date() }
      }),
    ]);
  }
}

OkHttp Interceptor — Auto JWT Refresh
The Retrofit client uses an Authenticator that automatically refreshes the access token when a 401 is received.

// TokenAuthenticator.kt
class TokenAuthenticator(private val tokenStore, private val authApi) : Authenticator {
  override fun authenticate(route, response): Request? {
    val refresh = tokenStore.getRefreshToken() ?: return null
    val newTokens = authApi.refresh(RefreshRequest(refresh)).execute()
    if (!newTokens.isSuccessful) { /* force logout */ return null }
    tokenStore.saveTokens(newTokens.body()!!)
    return response.request.newBuilder()
      .header('Authorization', 'Bearer '+newTokens.body()!!.access_token)
      .build()
  }
}

Direct Browser → Cloudinary Upload
Media uploads from the admin dashboard go directly to Cloudinary — not through the API server. This reduces server load and upload latency significantly.

// useCloudinaryUpload.js
async function uploadToCloudinary(file, preset = 'rs3-admin') {
  const formData = new FormData();
  formData.append('file', file);
  formData.append('upload_preset', preset);
  const res = await fetch(
    'https://api.cloudinary.com/v1_1/rs3films/auto/upload',
    { method: 'POST', body: formData }
  );
  const data = await res.json();
  return { secure_url: data.secure_url, public_id: data.public_id };
}
// Usage: admin drops image → upload to Cloudinary → get secure_url → send URL to API

08 — How App & Web Stay in Sync
This section explains exactly how each admin action in the dashboard propagates to the app in real time or near-real time.

Admin Action	How It Reaches App	Time to Live
Publish a new feed post	Redis cache invalidated. Next app feed request gets fresh data.	Immediate on next pull-to-refresh or app open.
Change splash image URL	app_content Redis cache invalidated. Cold start fetches new URL.	Live on user's next cold start (close + reopen app).
Edit T&C / Privacy text	app_content cache invalidated.	Live on next app launch (Room cache TTL: 24h).
Change coin value for a rule	point_rules Redis cache invalidated.	Effective for all registrations from that moment forward.
Add Level 4 to referral chain	Server reads updated active rules on next registration.	Immediate — next person who registers triggers L4 chain.
Toggle offer active/inactive	offers cache invalidated.	App sees change on next /offers poll or screen open.
Change coin_value_label	app_content cache invalidated.	App shows new label after Room cache refresh.
Enable maintenance_mode	app_content returned immediately on next app call.	App shows maintenance screen within seconds.
Change ambassador_threshold	app_content cache. App reads threshold from cached content.	Effective on next app launch.
Push admin notification	FCM message sent directly via Firebase Admin SDK.	Delivered in real time via push notification.

The Key Principle
The API never pushes data to the app unprompted (except push notifications via FCM).
The app always calls the API — on screen open, pull-to-refresh, or timer.
Redis caching means these calls return stale data until the admin action invalidates the cache.
For critical changes (maintenance mode, urgent announcement) — push notification + cache invalidation together = immediate reach.

10 — Security Checklist

API Security
•	All endpoints validate JWT on every request — no endpoint is publicly writable.
•	Passwords hashed with bcrypt (cost 12). Never logged, never returned in responses.
•	All DB queries via Prisma parameterized — zero SQL injection surface.
•	Rate limiting via Redis: OTP 3/phone/10min, login 10/IP/min, register 5/IP/hour.
•	CORS: API only accepts requests from the app bundle ID and dashboard domain.
•	Admin JWT in HttpOnly Secure SameSite=Strict cookie. Never in JS-accessible storage.
•	Selfie and media URLs are Cloudinary signed URLs with expiry. Not permanent public URLs.

Fraud Prevention
•	Coin rewards issued only after otp_verified = true. Never on form submission.
•	Self-referral: server checks referred_by_user_id !== new user's own ID.
•	Phone uniqueness enforced at DB level (UNIQUE constraint).
•	Offer over-claiming: max_claims checked with atomic DB increment. Race-condition safe.
•	coin_transactions append-only. No UPDATE or DELETE on this table ever.
•	coin_balance on users reconciled nightly from coin_transactions via cron job.
•	Admin manual coin adjustments require a reason field and are audit-logged.

Android Security
•	JWTs stored in EncryptedSharedPreferences (Android Keystore backed AES-256).
•	Certificate pinning: OkHttp pinned to API server's certificate. MITM protection.
•	Network Security Config: cleartext traffic disabled (enforces HTTPS everywhere).
•	ProGuard/R8 enabled in release builds: obfuscates class names.
•	No secrets in APK: all API keys are server-side or fetched at runtime.
